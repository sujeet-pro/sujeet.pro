Local markdown rules for `sujeet.pro`.

Use this file as the local entrypoint. For the full feature reference, read the installed upstream guide first:

- `node_modules/@pagesmith/core/skills/pagesmith-core-setup/references/markdown-guidelines.md`

For broader Pagesmith behavior, also read:

- `node_modules/@pagesmith/core/REFERENCE.md`

This file captures the repo-specific choices layered on top of those upstream docs.

## Core rules

- Use exactly one `# H1` per page.
- Keep heading depth sequential.
- Use fenced code blocks with a language identifier.
- Prefer markdown links like `[RFC 9110](https://www.rfc-editor.org/rfc/rfc9110)` over bare URLs.
- Use relative links for internal repo content.
- Do not add ad-hoc copy-button or code-block JS.
- Do not invent unsupported code block meta properties.

## Supported Pagesmith markdown features

The repo inherits the standard `@pagesmith/core` feature set:

- GFM: tables, strikethrough, task lists, footnotes, autolinks
- GitHub alerts: `NOTE`, `TIP`, `IMPORTANT`, `WARNING`, `CAUTION`
- Math via `$...$` and `$$...$$`
- Smart typography
- Auto heading anchors
- Expressive code block features

Use the upstream file for full syntax details. This local guide focuses on how to apply those features in this repo.

## Code blocks

Valid code block meta:

- `title="..."`
- `showLineNumbers`
- `startLineNumber=N`
- `mark={lines}`
- `ins={lines}`
- `del={lines}`
- `collapse={lines}`
- `wrap`
- `frame="..."`

Use these patterns:

````md
```ts title="cache.ts" showLineNumbers mark={3,8-10}
export function getCacheKey() {
  // ...
}
```
````

Avoid these mistakes:

- Use `mark={5}`, not bare `{5}`
- Use `title="file.ts"`, not `file=file.ts`
- Do not omit the language when using meta

## Frontmatter

Articles and blogs use the repo's `BaseFrontmatterSchema` shape:

```yaml
---
title: "Title"
description: "One-line summary"
publishedDate: 2026-03-15
lastUpdatedOn: 2026-03-20
tags: [topic-a, topic-b]
draft: true # Optional
---
```

Rules:

- Do not set `layout` or `category` in article or blog frontmatter.
- `title` and `description` are for SEO, cards, and feeds.
- The visible page title still needs a markdown `# H1`.

## Companion metadata checks

Markdown changes can require metadata changes too:

- Check `content/<section>/meta.json5` when the edit changes ordering, series membership, or listing expectations.
- Check `content/home.json5` when the entry is featured or linked from the homepage.
- Check `content/redirects.json5` when the slug or canonical path changes.
- Treat the markdown file and its coordinating JSON5 files as one change set.

## Diagrams in markdown

Use consecutive markdown images with `-light` and `-dark` suffixes. The Pagesmith pipeline automatically merges them into a themed `<figure>`:

```md
![System overview](./diagrams/overview-light.svg "High-level request flow and failure boundaries.")
![System overview](./diagrams/overview-dark.svg)
```

Notes:

- Use `-light.svg` and `-dark.svg` filenames generated by `diagramkit`.
- Always include useful alt text.
- Add a caption via the markdown title attribute on the light image: `![alt](src "caption")`.
- Light image always comes first, dark image immediately after.
- Do not use `<figure>`, `<img>`, or `<picture>` HTML for image or diagram embeds.

## Images

- Use `![alt](./assets/image.png)` for simple images.
- Use `![alt](./assets/image.png "caption text")` for images with captions.
- For themed light/dark pairs, place the light image first and dark image immediately after.
- Do not use `<figure>`, `<img>`, or `<picture>` HTML tags for images.

## Citations and footnotes

- Articles should cite every non-trivial factual claim.
- Blogs should cite non-obvious, external, numerical, or standards-based claims.
- Footnotes are good for dense references that would clutter the sentence.
- Inline links are good when the source naturally fits the prose.

Example footnote:

```md
The event loop drains microtasks before returning to the task queue[^1].

[^1]: [WHATWG HTML - Event loops](https://html.spec.whatwg.org/multipage/webappapis.html#event-loops)
```

## Validator expectations

The built-in Pagesmith validators matter for this repo:

- `linkValidator`
- `headingValidator`
- `codeBlockValidator`

Common failure modes:

- bare URLs
- skipped heading levels
- missing code block languages
- unsupported meta properties

If validation errors appear, use:

- `node_modules/@pagesmith/core/skills/pagesmith-core-setup/references/errors.md`

## Practical style choices

- Use tables for trade-offs and comparisons.
- Use alerts sparingly and intentionally.
- Use diagrams when prose alone makes the mechanism too hard to follow.
- Prefer shorter paragraphs than you would in print because most readers skim first.
